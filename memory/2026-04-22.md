## Browser Automation for HubSpot CRM

Successfully built end-to-end browser automation for [[Alert Pro]] HubSpot workflows using Playwright. #win #project/alert-pro

**Key learnings:**
- Persistent browser profiles work (`~/siro-browser-profile`) - OAuth sessions persist between runs
- SPA navigation: never use `waitForLoadState('networkidle')` - React apps never go idle. Use `waitForSelector()` for specific elements or `waitForTimeout()` with `domcontentloaded`
- HubSpot's filter dropdowns use checkboxes and don't auto-close. Must explicitly close them (Escape key safest)
- Rich text editors: use `page.keyboard.type()` after clicking into field, not `.fill()` or `.type()` on locator (prevents duplication)
- Modal/popup dismissal: run after every major action to prevent UI blocking

**Working flow:**
1. Navigate to accounts page → select Alert-Pro Solutions portal
2. Navigate to Contacts page
3. Clear existing filters, apply new filter
4. Close dropdown (Escape key)
5. Hover first contact → click Preview button
6. Click Email button in preview panel
7. Fill subject + body (preserves signature)
8. Never click Send

**Technical patterns:**
- Headed mode with `slowMo: 800` for visibility
- `DISPLAY=:0` for WSLg on Windows
- Screenshot at every step for debugging
- Multiple selector fallbacks (text, aria-label, role, data-test-id)
- Popup dismissal helper runs periodically

**Blockers encountered:**
- Filter dropdown blocking contact interaction if not closed properly
- Double text input when using both `.fill()` and `.type()` on same field
- "Lead status" filter persisting after "Clear all" - must clear individually

**Scripts created:**
- `~/scripts/browser.mjs` - Persistent profile launcher
- `~/scripts/hubspot-navigate.mjs` - Full CRM workflow automation
- Screenshots saved to `~/scripts/screenshots/`

## HubSpot Automation Debugging Session

Spent ~90 minutes iteratively debugging the HubSpot email draft automation. #learning #project/alert-pro

**Problems encountered:**
1. **Filter dropdown not closing** - clicking page elements was accidentally opening/applying new filters that wiped contact list to 0 records. Solution: simplified to only Escape key + pill re-click, removed aggressive click-anywhere strategies
2. **Preview button not found** - CSS selectors weren't matching the button that appeared on hover. Solution: switched to `page.getByText('Preview')` and `page.getByRole('button')` which are more reliable
3. **Contact name click navigated away** - clicking the contact name link opened full record page instead of preview panel. Solution: only click the Preview button, not the name
4. **Filter pills missing** - script expected "Contact owner" filter to already exist. Solution: added "Add filter" → "Contact owner" creation step if pill doesn't exist
5. **Preview panel closed immediately** - taking screenshot or looking for Email button was dismissing the panel. Solution: increased wait from 2s → 3s and used fullPage screenshot

**Final working flow (165 unassigned contacts):**
- Hover second contact (Josh Craft from Alleghany Highlands Public Schools)
- Click Preview button → side panel opens and stays open
- Click Email button in preview
- Type subject + body with `page.keyboard.type()`
- Insert signature (tried but Insert menu didn't have Signature option - skipped gracefully)
- Add spacing (2x Enter)
- Click Documents → click "Alyssa's Law Compliance - Alert Pro Solutions.pdf" title
- Document attaches to email body
- Draft left unsent

**Key debugging pattern:** Screenshot at every step + detailed console logging + graceful failures (warnings instead of fatal errors when optional features like signature insertion fail)

**Lesson:** When CSS selectors fail on dynamic UI, prefer Playwright's semantic locators (`getByRole`, `getByText`) over complex CSS. They're more resilient to DOM changes.
